## Executive Summary

Performance testing of the metrics-service collection and rollup pipeline using real AWX data generated by metrics-utility generators. Tests measure each component separately with time and memory metrics.

**Key Findings:**

- ✓ **Production-scale validated:** 11.4M raw events processed successfully
- ✓ **Collection phase:** ~82 minutes to process 11.4M raw events into 72 hourly collections (43.13 MB)
- ✓ **Rollup phase:** Sequential execution (2.47s) more memory-efficient than parallel (1.81s)
- ✓ **Memory efficiency:** Sequential rollup uses 82% less memory (48 MB vs 265 MB)
- ✓ **Cache warming:** Sequential rollup 60% faster due to warm database cache
- ✓ **Real-world testing:** Uses actual AWX database populated by metrics-utility generators

## Cache Warming Effect

### Rollup Task Performance

| Execution Mode | Rollup Duration | Cache Status | Performance |
|----------------|-----------------|--------------|-------------|
| **Parallel** | 1,805.08 ms (1.81s) | Cold cache | Baseline |
| **Sequential** | 1,126.01 ms (1.13s) | Warm cache | **60% faster** |

**Cause:** Hourly collection tasks load data into PostgreSQL shared buffers; rollup benefits from warm cache.

**Implication:** Sequential execution order (collectors → rollup) provides consistent cache warming benefits.

## Production Recommendations

### Execution Strategy

**Recommended: Sequential Execution**

```python
# Sequential execution with task ordering
tasks = [
    'collect_job_host_summary_hourly',  # Warms cache
    'collect_host_metrics_hourly',       # Warms cache
    'collect_main_host_hourly',          # Warms cache
    'daily_metrics_rollup',              # Benefits from warm cache
    'daily_anonymize_and_prepare',
    'send_anonymized_to_segment',
    'full_process_anonymize',
    'cleanup_metrics_data',
]

# Run sequentially
for task in tasks:
    execute_task(task)
```

**Benefits:**
- 60% faster rollup due to cache warming
- 82% less memory usage (48 MB vs 265 MB)
- Predictable resource consumption
- Better for containerized deployments

**Trade-offs:**
- 36% slower total time (2.47s vs 1.81s) - negligible compared to 82-minute collection phase
- Sequential dependencies (one task blocks others)

### Resource Requirements

| Deployment Size | Raw Events | Collection Time | Rollup Time | Memory (Rollup) | Disk (Collections) |
|-----------------|------------|-----------------|-------------|-----------------|-------------------|
| Small | <1M | ~7 min | <1s | <10 MB | <5 MB |
| Medium | 1M-10M | ~30-70 min | 1-2s | 10-50 MB | 5-50 MB |
| **Large (tested)** | **11.4M** | **~82 min** | **2.5s** | **48 MB** | **43 MB** |
| X-Large | 50M+ | ~6-7 hours | 3-5s | 100-200 MB | 150-250 MB |

### Optimization Priorities

**High Impact:**
1. ✓ **Use sequential execution** (5.5x memory savings)
2. ✓ **Maintain task order** (60% rollup speedup from cache warming)
3. ✓ **Monitor collection phase** (99.95% of total time)

**Medium Impact:**
4. Optimize collection queries (reduce 82-minute collection time)
5. Add database indexes on aggregation columns
6. Implement incremental collections (only new events)

**Low Impact:**
7. Rollup optimization (only 0.05% of total pipeline time)
8. Parallel execution (saves 0.7s but uses 5.5x memory)

**Don't Do:**
- ✗ Optimize rollup before collection (wrong bottleneck)
- ✗ Use parallel execution in containers (memory overhead)
- ✗ Run rollup independently (loses cache warming benefit)
